Let us explain how we prevent spam in the "Isolated Mailboxes"

First let's have a look at this simple mail conversation.

Mail Conversation

  • mail.amazon.com Connecting to mail.domboxmail.com with its IP address
  • Dombox : 220 mail.domboxmail.com Dombox SMTP Service Ready
  • Amazon : HELO mail.amazon.com
  • Dombox : 250 Hello, nice to meet you, mail.amazon.com
  • Amazon : MAIL FROM: <jeff@amazon.com>
  • Dombox : 250 OK
  • Amazon : RCPT TO: <test123$amazon.co.uk@domboxmail.com>
  • Dombox : 250 OK
  • Amazon : DATA
  • Dombox : 354 End data with <CRLF>.<CRLF>
  • Amazon : From: Jeff Bezos <jeff@amazon.com>
  • Amazon : To: Test User <test123$amazon.co.uk@domboxmail.com>
  • Amazon : Date: Fri, 01 January 2015 16:02:43 -0500
  • Amazon : DKIM-Signature: a=rsa-sha1; q=dns; s=selector123; d=amazon.ca; ……….
  • Amazon : Subject: Merry Christmas
  • Amazon : Merry Christmas and a Happy New Year.
  • Amazon : Regards,
  • Amazon : Jeff Bezos
  • Amazon : .
  • Dombox : 250 OK, message accepted for delivery: queued as 12345
  • Amazon : QUIT
  • Dombox : 221 Bye
  • "Envelope Part" represented in "Normal Font". This part is intended for mail handling servers.
  • "Message Part" represented in "Italics Font". This is the part that get displayed to the user.

Detailed Description

  • mail.amazon.com Connecting to mail.domboxmail.com with its IP address
  • At this stage we got the sending server IP address. Let's call this IP address 100.100.100.100
  • Let's save that IP address in this variable. SENDING_SERVER_IP_ADDRESS
  • SENDING_SERVER_IP_ADDRESS = 100.100.100.100
  • Dombox : 220 mail.domboxmail.com Dombox SMTP Service Ready
  • This part says, our mail server is ready to listen to the Conversation
  • Amazon : HELO mail.amazon.com
  • Amazon server, introduces itself saying "Hello, My name is Amazon"
  • Dombox : 250 Hello, nice to meet you, mail.amazon.com
  • Our server responds politely
  • Amazon : MAIL FROM: <jeff@amazon.com>
  • This part is known as "Envelope From"
  • The sending server claims that the mail is coming from jeff@amazon.com
  • Hold On! We need to make sure that claim is true
  • So we are gonna ask our first question to the "Envelope Domain". Here amazon.com is the "Envelope Domain"
  • Question 1: Hello "Envelope Domain i.e. amazon.com", we have a sender who claims the mail is from your domain. Could you give us your "Authorized IP Addresses" for your outgoing mails?
  • Answer : Sure. Here is our "Authorized IP Addresses" => "100.100.100.100, 150.150.150.150, 200.200.200.200"
  • If the SENDING_SERVER_IP_ADDRESS found in the "Authorized IP Addresses", then the result is "Pass"
  • If the SENDING_SERVER_IP_ADDRESS not found in the "Authorized IP Addresses", then the result is "Fail"
  • If amazon.com returns empty response, then SENDING_SERVER_IP_ADDRESS is neither Authorized nor Unauthorized. So the result is "Neutral"
  • Dombox : 250 OK
  • Our server acknowledges the "MAIL FROM" command issued by the sender
  • Amazon : RCPT TO: <test123$amazon.co.uk@domboxmail.com>
  • The sending server states the recipient. In this case test123$amazon.co.uk@domboxmail.com
  • Hold On! That doesn't seem right.
  • "Envelope Domain" is "amazon.com". i.e. MAIL FROM: <jeff@amazon.com>
  • However the "Dombox Domain" is "amazon.co.uk". i.e. RCPT TO: <test123$amazon.co.uk@domboxmail.com>
  • amazon.co.uk is the House owner. But amazon.com is trying to enter the house.
  • Maybe amazon.com is an "Alias Domain" for amazon.co.uk. So we are gonna ask our second question to the "Dombox Domain".
  • Note: We are gonna ask this question two times. To distinguish, let's call this question as 2(a).
  • Question 2(a): Hello "Dombox Domain i.e. amazon.co.uk", Could you give us your "Alias Domains" for your outgoing mails?
  • Answer : Sure. Here is our "Alias Domains" => "amazon.com:r+b example.com:s+m example.net"
  • Let's format the response.
  • amazon.com:r+b
  • example.com:s+m
  • example.net
  • So amazon.co.uk says those three domains are their "Alias Domains". Meaning, those "Alias Domains" can send mails to the "amazon.co.uk" Dombox.
  • Note: "amazon.co.uk" can set rules for each "Alias Domain". That's what you see after the "colon" character
  • Rules
  • {Domain}:{Relaxed or Strict}+{Envelope Mode or Message Mode or Both}
Relaxed (r) Exact domain and its subdomains are allowed (Default)
Strict (s) Exact domain only allowed.
Envelope Mode (e) Domain is allowed only in the “Envelope From” (Default)
Message Mode (m) Domain is allowed only in the “Message From”
Both Mode (b) Domain is allowed in “Envelope From” as well as “Message From”
  • So, “example.net” is equivalent to “example.net:r+e”
  • Result
  • If "Dombox Domain" and "Envelope Domain" are same, then we don't have to ask this question at all. This is called "Direct Pass"
  • In our case, this is "Indirect Pass".
  • Since "amazon.co.uk" whitelisted "amazon.com" via DNS, we allow this mail.
  • Note: amazon.com has "Both Mode (b)". That means it's allowed in both "Envelope From" as well as "Message From"
  • If the result is "Fail", then the mail will be rejected for that particular recipient
  • Dombox : 250 OK
  • Our server acknowledges the "RCPT TO" command issued by the sender
  • Amazon : DATA
  • Amazon indicates that they are about to begin the original message. i.e. Message Part
  • Dombox : 354 End data with <CRLF>.<CRLF>
  • Our server tells amazon to go ahead using the code 354
  • Amazon : From: Jeff Bezos <jeff@amazon.com>
  • Amazon : To: Test User <test123$amazon.co.uk@domboxmail.com>
  • Amazon : Date: Fri, 01 January 2015 16:02:43 -0500
  • Amazon : DKIM-Signature: a=rsa-sha1; q=dns; s=selector123; d=amazon.ca; ……….
  • Amazon : Subject: Merry Christmas
  • Amazon : Merry Christmas and a Happy New Year.
  • Amazon : Regards,
  • Amazon : Jeff Bezos
  • Amazon : .
  • We got the whole original message now.
  • We are gonna ask three more questions here.
  • Before that, let's understand the terms
  • Amazon : From: Jeff Bezos <jeff@amazon.com>
  • This part is called "Message From"
  • amazon.com is the "Message Domain" here
  • Amazon : DKIM-Signature: a=rsa-sha1; q=dns; s=selector123; d=amazon.ca; ……….
  • This part is called "Signature"
  • amazon.ca is the "Signature Domain" here
  • Question 2(b): We already know amazon.com is an "Alias Domain" for "amazon.co.uk" via Question 2(a). But is it allowed in "Message From"?
  • "amazon.com:r+b example.com:s+m example.net"
  • Since we see "Both Mode (b)" for amazon.com, the answer is "Yes". So we are gonna allow this mail.
  • Question 3: Hello "Signature Domain (i.e. amazon.ca)", could you give us your "Public Key" for verifying "Digital Signature"?
  • Answer : Sure. Here is our "Public Key" => "......."
  • We get the "Public Key" from "amazon.ca" and verify the signature.
  • Note: The signature verification would result in "Fail" even if a single white space added in the message during transit.
  • So "Digital Signature" not only helps us in "Authentication", but also helps us to prevent "Message Modification" during transit
  • Question 4: Hello "Message Domain (i.e. amazon.com)", could you give us your misalignment "Policy"?
  • Answer : Sure. Here is our misalignment policy => "None"
  • Amazon : MAIL FROM: <jeff@amazon.com>
  • Amazon : From: Jeff Bezos <jeff@amazon.com>
  • Amazon : DKIM-Signature: a=rsa-sha1; q=dns; s=selector123; d=amazon.ca; ……….
  • Envelope Domain: amazon.com
  • Message Domain: amazon.com
  • Signature Domain: amazon.ca
  • Clearly, there is a mismatch
  • "Signature Domain" not aligned with the "Message Domain"
  • Many spammers use the spoofing method to send Phishing mails.
  • Companies like PayPal had been a major victim of Phishing mails in the past.
  • Keep in mind, "Message Part" is the one that get displayed to the user. That's why we need to make sure "Message Domain" is not a spoofed one.
  • So our Subject of Interest is the "Message Domain"
  • If the "Message Domain" doesn't match with "Envelope Domain" or "Signature Domain", then we need to apply the policy given by the "Message Domain"
  • The Policy can be one of the following three things. None, Quarantine, Reject
  • None - Do whatever you want, Quarantine - Put in the spam folder, Reject - Reject the mail immediately
  • If "PayPal" says "Reject", that means they are telling us to not accept mail from anyone else
  • Result
  • This mail will be accepted because the "Message Domain i.e. amazon.com" has "No Objection" due to the "None" policy
  • Dombox : 250 OK, message accepted for delivery: queued as 12345
  • Our server informs the sender that the mail is accepted.
  • Amazon : QUIT
  • Amazon says "Ok. I'm done"
  • Dombox : 221 Bye
  • Our server finishes the conversation with the 221 code

Terminology

Let's have little bit of technical terms here

The one where we checked for "Authorized IP Address" is called "Authorization Layer (SPF)"

The one where we checked for "Alias Domains" is called "Alias Layer (SAD)"

The one where we checked for "Signature" is called "Authentication Layer (DKIM)"

The one where we checked for "mislignment policy" is called "Alignment Layer (DMARC)"

Finally we also check whether the mail is encrypted or not. This is called "Encryption Layer (TLS)"

Note: Our system relies on already existing standards like TLS, SPF, DKIM and DMARC. So we are not trying to reinvent the wheel for those checks.

However, for "Alias Layer", we had to introduce our own standard. We call it "Sender Alias Domains (SAD)"

Rules

The following layers must be passed for the mail to be accepted.
